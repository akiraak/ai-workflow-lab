# B5: TDD Bug Fix パターンの効果測定

## 仮説

TDD Bug Fix パターン（失敗テスト先行 → 修正 → テスト通過確認）は、直接修正パターンと比べて修正成功率が **15% 以上** 高い。

- 元調査: [09-task-bug-fix.md](/experiments/claude-code-prompting/09-task-bug-fix) — TDD Bug Fix パターンが推奨されていたが、定量データなし

## 実験条件

| 項目 | 値 |
|------|-----|
| 独立変数 | 修正アプローチ（TDD / 直接修正）× バグ（bug1 / bug2）= 4 条件 |
| モデル | claude-sonnet-4-20250514 |
| CLAUDE.md | 共通（事実形式、30 行） |
| 試行数 | 各条件 5 回（合計 20 回） |

### 2 つのバグ

| バグ | 対象 | 症状 | 原因 |
|------|------|------|------|
| bug1 | discount.ts | `applyDiscount(1000, 110)` → `-100`（負の価格） | percentage > 100 のバリデーション欠落 |
| bug2 | cart.ts | `calculateTotal([])` → TypeError | 空配列チェックと reduce 初期値の欠落 |

### 2 つのアプローチ

| 条件 | 手順 |
|------|------|
| tdd | 失敗テスト作成 → 失敗確認 → 修正 → テスト通過確認 |
| direct | バグを修正 → テストで確認 |

## 判定: 仮説不支持

両手法とも修正成功率 100%（テスト通過・回帰なし・バグ修正すべて完全）で差が出なかった。

## 考察

- **天井効果**: 明確なバリデーションバグ（範囲チェック欠落、空配列未処理）に対しては、Claude はどちらの手法でも確実に修正できる。差を検出するにはより複雑なバグが必要
- **TDD の主効果はテストカバレッジの安定**: 新規テスト追加率が TDD 100% vs direct 50%。TDD ではプロンプトで明示的にテスト作成を指示するため、テストが常に追加される
- **direct のほうが効率的**: ツール呼出数が約 15% 少ない（7.9 vs 9.1）。TDD の失敗テスト作成→確認のステップ分だけオーバーヘッドが発生
- **使い分けの指針**: テストカバレッジの安定が重要な場面では TDD パターン、単純なバグの迅速な修正では直接修正が適切
