プロジェクト構造を確認した結果、以下の実装計画を提案します。

## 変更するファイル一覧

1. `src/order.ts` - createOrder 関数に通知機能を追加
2. `tests/order.test.ts` - 新規作成、注文作成時の通知テストを追加

## 各ファイルの変更内容

### src/order.ts の変更
- `notification.ts` と `user.ts` からの import を追加
- `createOrder` 関数内で以下を実装：
  - `getUserById(userId)` でユーザー情報を取得
  - ユーザーが見つからない場合のエラーハンドリング
  - 注文内容のサマリーテキスト生成（アイテム名と合計金額）
  - `sendNotification(email, subject, body)` を呼び出し
- 戻り値型を `Order` から `{ success: boolean; data?: Order; error?: string }` に変更してエラーケースに対応

### tests/order.test.ts の変更（新規作成）
- 必要な依存関係のimport（vitest、対象関数、テストデータ準備用関数）
- `describe("createOrder")` ブロック内に以下のテストケース：
  - 正常ケース：注文作成と通知送信の確認
  - 異常ケース：存在しないユーザーIDでの注文作成
  - 通知内容の確認（件名、本文の形式）
- テスト前後でのデータクリーンアップ（`clearOrders()`, `clearUsers()`, `clearNotifications()` の呼び出し）

## 実装の順序

1. **テストファイルの作成**（TDDアプローチ）
   - `tests/order.test.ts` を作成
   - 期待する動作のテストケースを記述

2. **createOrder 関数の修正**
   - エラーハンドリングを含む通知機能を実装
   - テストが通るように段階的に実装

3. **動作確認**
   - `npx vitest run tests/order.test.ts` でテスト実行
   - 全テストが通ることを確認

## リスクや注意点

### 🔴 高リスク
- **破壊的変更**: `createOrder` の戻り値型を変更すると既存の呼び出し元に影響する可能性
- **依存関係の循環**: order.ts → user.ts の依存が追加されるため、循環参照に注意

### 🟡 中リスク
- **エラーハンドリング**: 存在しないユーザーIDでの注文作成時の適切な処理
- **テストデータ管理**: 複数テスト間でのデータ状態管理

### 🟢 低リスク
- **通知内容の形式**: サマリーテキストの生成ロジック
- **型安全性**: 既存のインターフェースとの整合性

### 対応策
1. 既存コードへの影響を最小化するため、まず戻り値型は変更せず、通知送信が失敗してもエラーをthrowしない実装を検討
2. テスト実行前に必ずデータクリア関数を呼び出してテスト間の独立性を確保
3. ユーザーが存在しない場合は通知を送信せずに警告ログのみ出力する選択肢も検討

この計画に基づいて段階的に実装を進めることで、安全かつ確実に機能を追加できます。
