# B4: Plan mode の初回成功率測定

## 仮説

Plan mode を使用すると、中規模タスク（3-5 ファイル変更）の初回成功率が **20% 以上** 向上する。

- 元調査: [03-plan-mode.md](/experiments/claude-code-prompting/03-plan-mode)、[10-task-new-feature.md](/experiments/claude-code-prompting/10-task-new-feature) — ある報告では 37% 向上とされるが、再現性は不明

## 実験条件

| 項目 | 値 |
|------|-----|
| 独立変数 | Plan mode（なし / あり）× モデル（Sonnet / Opus）= 4 条件 |
| タスク | 注文完了時のメール通知機能追加（3 ファイル変更） |
| CLAUDE.md | 共通（事実形式、30 行） |
| 試行数 | 各条件 5 回（合計 20 回） |

### 4 条件

| 条件 | Plan mode | モデル |
|------|-----------|--------|
| no-plan × sonnet | なし（直接実装） | claude-sonnet-4-20250514 |
| plan × sonnet | あり（計画→実装の 2 段階） | claude-sonnet-4-20250514 |
| no-plan × opus | なし | claude-opus-4-20250514 |
| plan × opus | あり | claude-opus-4-20250514 |

### 判定指標

| 指標 | 定義 |
|------|------|
| tests_pass | テストが全通過するか |
| typecheck_pass | `tsc --noEmit` が通るか |
| lint_pass | ESLint エラーなしか |
| calls_notify | `createOrder` が `sendNotification` を呼んでいるか |
| uses_email | 通知の宛先がユーザーの email か |
| subject_id | 件名に orderId が含まれるか |
| test_notify | テストが通知送信を検証しているか |
| first_success | 上記すべてが通るか |

## 判定: 仮説不支持

全 4 条件でテスト通過率 0%、初回成功率 0%。Plan mode の効果を比較できなかった。

## 考察

- **床効果が発生**: tests_pass が全条件 0% のため、Plan mode あり/なしの差を検出できなかった。機能実装自体（通知呼出、メール使用など）は全条件 100% 成功しており、テスト自体の設計難度が高すぎた可能性がある
- **Plan mode はツール呼出数を増加させる**: sonnet で +2.0（+11%）、opus で +1.2（+9%）。計画フェーズのオーバーヘッドが加わるため、成功率が同等なら直接実装のほうが効率的
- **Opus は Sonnet より効率的**: ツール呼出数が約 27% 少ない（平均 14.4 vs 19.6）。同じ結果をより少ないステップで達成
- **plan × sonnet で subject_id が 60% に低下**: 計画段階で要件の解釈ブレが発生し、件名のフォーマットが仕様から逸脱。Plan mode が必ずしも精度を上げるわけではない

### テスト通過率 0% の原因

型チェック・lint・機能実装はすべて成功しており、「通知が送信されることを確認するテスト」の設計だけが失敗している。通知のモック化が困難なテスト構造だった可能性が高く、より適切な難易度のタスクでの再実験が必要。
