# Phase 4B 実証実験 統合レポート

## 概要

Phase 1〜3 の Web 調査で収集したコミュニティ知見・ベストプラクティスのうち、定量データが存在しなかった 6 つのコア仮説を実証実験（B1〜B6）で検証した。全 115 試行の結果を統合分析し、実践的な提言をまとめる。

**実験期間**: 2026-04-07 〜 2026-04-09
**総試行数**: 115（B1: 20, B2: 20, B3: 21, B4: 20, B5: 20, B6: 14）
**使用モデル**: Claude Sonnet（主）、Claude Opus（B4 のみ比較対象）

### B3: CLAUDE.md 行数と遵守率の関係

**条件**: short（23 行）7 試行 / medium（96 行）7 試行 / long（247 行）7 試行
**タスク**: B1 と同一（メールバリデーション関数の追加）

#### 項目別遵守率

| 項目 | short | medium | long |
|------|-------|--------|------|
| r1〜r5 | 100% | 100% | 100% |
| r6 ファイルサイズ | 57% | 29% | 86% |

**タスク**: 注文完了時のメール通知機能追加（3 ファイル変更）

|------|-----------|-----------|------|---------|-----------|--------|-----------|---------|-------------------|

### B5: TDD Bug Fix パターンの効果測定

**条件**: tdd / direct × bug1（割引率バグ）/ bug2（カート計算バグ）= 4 条件（各 5 試行）
**タスク**: 既知バグの修正

#### 結果サマリ

| 条件 | テスト通過 | 回帰なし | バグ修正 | 新規テスト数 (平均) | ツール呼出数 (平均) |
|------|-----------|---------|---------|-------------------|-------------------|
| tdd × bug1 | 100% | 100% | 100% | 2.2 | 8.8 |
| tdd × bug2 | 100% | 100% | 100% | 1.0 | 9.4 |
| direct × bug1 | 100% | 100% | 100% | 1.0 | 7.8 |
| direct × bug2 | 100% | 100% | 100% | 0.6 | 8.0 |

#### 判定: **仮説不支持**

両手法とも成功率 100%（テスト通過・回帰なし・バグ修正すべて完全）で差が出なかった。

**副次的知見**:
- TDD はテストカバレッジが安定する（新規テスト数: tdd 1.6 vs direct 0.8）
- TDD は bug1 で常に 2〜3 件のテストを追加、direct は 0〜3 件でばらつきが大きい

- 明確なバリデーションバグでは、どちらの手法でも Claude は確実に修正できる

**条件**: no-hooks / hooks（各 7 試行）
**タスク**: B1 と同一（メールバリデーション関数の追加）

#### 結果サマリ

| 条件 | ESLint エラー (平均) | ESLint エラー (合計) | テスト通過率 | 型エラー |
|------|---------------------|---------------------|-------------|---------|
| no-hooks | 1.0 | 7 | 100% | 0 |
| hooks | 0 | 0 | 100% | 0 |

#### no-hooks の ESLint エラー詳細

| 試行 | ESLint エラー数 |
|------|----------------|
| 1 | 3 |
| 2 | 0 |
| 3 | 2 |
| 4 | 2 |
| 5 | 0 |
| 6 | 0 |

#### 判定: **仮説支持**

Hooks 条件は全 7 試行で ESLint エラー 0 件（完全排除）。no-hooks 条件は 7 試行中 3 試行でエラーが発生（発生率 43%、平均 1.0 件）。Hooks は lint エラーを **100% 削減**し、基準の 50% を大幅に超えた。

テスト通過率・型エラーは両条件とも同一で、Hooks は lint/フォーマットに限定した効果を持つ。

## 3. 横断分析

### 3-1. 仮説検証の総括

| # | 仮説 | 判定 | 実測効果 | 基準 |
|---|------|------|---------|------|
| B1 | 事実形式 > 命令形式 | **不支持** | +4%（+0.4 点） | 10%+ |
| B2 | 否定形の違反率 2 倍 | **不支持** | 差なし（−0.2 点） | 違反率 2 倍 |
| B3 | 行数増加 → 遵守率低下 | **不支持（逆転）** | long が最高（9.86） | 単調減少 |
| B4 | Plan mode で成功率 +20% | **不支持** | 全条件 0%（天井効果） | 20%+ |

### 3-2. なぜ差が出なかったのか — 構造的要因の分析

#### 天井効果（Ceiling Effect）

B1〜B3 のコア規約（r1〜r5, r8〜r10）は全条件で 100% 遵守。B5 は全条件で成功率 100%。Claude は明確な規約・明確なバグに対して、指示の書き方に関係なく高い遵守率を示す。差が生じたのは「暗黙的・周辺的な規約」（r6 ファイルサイズ、r7 JSDoc）に限定される。

#### タスク難易度と差の検出力

| 実験群 | タスク難易度 | ベースライン成功率 | 差の検出 |
|--------|-----------|------------------|---------|
| B1〜B3 | 低（単一関数追加） | 89〜99% | 困難（天井効果） |
| B4 | 中（3 ファイル変更） | 0%（テスト）/ 100%（その他） | 困難（床効果+天井効果） |
| B5 | 中（バグ修正） | 100% | 困難（天井効果） |
| B6 | 低（単一関数追加） | 57〜100% | 可能（lint に限定） |

簡単なタスクでは条件間の差が現れにくい。B4 のテスト通過率 0% は逆方向の「床効果」で、タスク設計がモデルの能力上限を超えていた可能性がある。

#### 決定論的 vs 助言的メカニズム

B6（Hooks）のみが仮説を支持した理由は明確: Hooks は決定論的（deterministic）に動作する。PostToolUse フックが Write/Edit 後に必ず `eslint --fix` と `prettier --write` を実行するため、Claude の判断に依存しない。一方、CLAUDE.md の記述スタイル（B1〜B3）は助言的（advisory）であり、Claude の解釈に委ねられる。

この結果は Phase 1〜3 調査の知見「CLAUDE.md は助言的、Hooks は決定論的」（04-slash-commands.md）を定量的に裏付けている。

### 3-3. 差が検出された周辺指標の分析

差が出なかった「主要指標」とは別に、副次的な指標で興味深いパターンが観察された。

#### r6（ファイルサイズ制約）の条件間比較

| 実験 | 条件 A | 条件 B | 条件 C |
|------|--------|--------|--------|
| B1 | factual 40% | imperative 20% | — |
| B2 | positive 20% | negative 30% | — |
| B3 | short 57% | medium 29% | long **86%** |

r6 は「生成ファイルを 200 行以下に抑える」という制約で、最も遵守率のばらつきが大きい。B3-long が突出して高いことから、**具体例やアンチパターンを含む詳細な記述が周辺的規約の遵守率を押し上げる**ことが示唆される。

#### ツール呼出数の比較

| 実験 | 条件 | 平均ツール呼出数 |

| B4 | no-plan × sonnet | 18.6 |

| B4 | plan × opus | 15.0（+9%） |

Plan mode と TDD はいずれもツール呼出数を増加させる（コストが上がる）。成功率に差がない場合、効率面で直接的アプローチが有利。

## 4. 実践的な提言

### 4-1. 高優先度（実験で裏付けられた推奨）

#### Hooks を積極的に活用する

B6 の結果から、PostToolUse Hooks による lint/format の自動実行は最もコスト対効果が高い品質改善手法。CLAUDE.md での「lint を実行してください」という助言的指示よりも、Hooks で決定論的に強制するほうが確実。

**推奨設定例**:
```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{
        "command": "npx eslint --fix \"$CLAUDE_FILE_PATH\" && npx prettier --write \"$CLAUDE_FILE_PATH\""
      }]
    }]
  }
}
```

#### CLAUDE.md は詳しく書いてよい

B3 の結果から、247 行の長い CLAUDE.md が 23 行の短いものより高い遵守率を示した。「コンテキストを節約するために短くすべき」という懸念は、少なくとも 250 行程度までは不要。具体例・アンチパターン・ファイル別解説を含めることで、周辺的規約（ファイルサイズ、JSDoc など）の遵守率が向上する。

### 4-2. 中優先度（差は小さいが参考になる知見）

#### CLAUDE.md の書き方は好みで選んでよい

B1・B2 の結果から、事実形式/命令形式、肯定形/否定形の違いは実用上無視できる。チームで統一する場合は可読性を基準に選べばよい。

#### TDD パターンはテストカバレッジを安定させる

B5 で成功率に差はなかったが、TDD は新規テスト作成数が安定する（常に 1〜3 件）。テストカバレッジの安定が重要な場面では TDD パターンのプロンプトが有用。ただし、明確なバグ修正では直接修正のほうが効率的（ツール呼出 15% 削減）。

### 4-3. 低優先度（追加検証が必要）

#### Plan mode の効果は未確定

B4 では全条件でテスト通過率 0% となり、Plan mode の効果を測定できなかった。より適切な難易度のタスクで再実験が必要。ただし、ツール呼出数の増加（+10%）は確認されており、簡単なタスクでは Plan mode のオーバーヘッドが顕在化する。

## 5. 制約と今後の課題

### 実験の制約

1. **タスク難易度の偏り**: B1〜B3・B6 は同一の単純タスク（関数追加）を使用。難易度が低すぎて天井効果が発生し、条件間差の検出が困難だった
2. **サンプルサイズ**: 各条件 5〜10 試行で、統計的検定力が限定的。特に B3（各 7 試行）と B6（各 7 試行）は 3 条件・2 条件での比較にはやや少ない
3. **単一プロジェクト**: sample-project / sample-project-clean の 2 種のみ。プロジェクト規模・言語・フレームワークの違いによる影響は未検証
4. **B4 のテスト設計**: テスト通過率 0% は実験タスクの設計問題の可能性が高い。通知のモック化が難しいテスト構造だった可能性がある

### 今後の追加検証候補

| テーマ | 目的 | 優先度 |
|--------|------|--------|
| 中〜高難易度タスクでの B1〜B3 再実験 | 天井効果を回避し、書き方の差を検出する | 高 |
| 適切なテスト設計での B4 再実験 | Plan mode の本来の効果を測定する | 高 |
| 複数プロジェクトでの B6 追試 | Hooks の効果の汎用性を確認する | 中 |
| CLAUDE.md 行数の上限探索 | 300 行、500 行でも遵守率が維持されるか | 中 |
| より複雑なバグでの B5 再実験 | 複数ファイルにまたがるバグでの TDD 効果 | 低 |

## 6. 結論

Phase 4B の実証実験で最も明確に得られた知見は次の 3 点である:

1. **決定論的メカニズム（Hooks）は助言的メカニズム（CLAUDE.md）より確実に品質を保証する** — CLAUDE.md の記述スタイルを工夫するよりも、Hooks で lint/format を自動実行するほうが効果が大きい（B6: lint エラー 100% 削減）

2. **CLAUDE.md は「短くすべき」ではなく「具体的に書くべき」** — 長い CLAUDE.md（247 行）が最高の遵守率を示した。コンテキスト消費よりも情報の具体性が重要（B3: long 9.86 > short 9.43）

3. **Claude は明確な規約に対して高いベースライン遵守率を持つ** — 記述スタイル（事実 vs 命令、肯定 vs 否定）の違いは実用上無視できるレベル。差が出るのは暗黙的・周辺的な規約に限られる（B1: +4%、B2: 差なし）

これらの知見は、Phase 4A のガイドライン改訂（特に A1: CLAUDE.md テンプレート、A5: 検証手段ガイド）に直接反映すべきである。
