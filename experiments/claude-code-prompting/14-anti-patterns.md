# アンチパターン（避けるべき指示方法）

## 概要

Claude Code に指示をする際に避けるべきパターンと、よくある失敗パターンを公式ドキュメント・コミュニティの知見から整理する。各パターンについて「なぜ避けるべきか」「代わりにどうすべきか」を示す。

---

## 避けるべき指示方法

### 1. Kitchen Sink Session（ごった煮セッション）

**パターン**: 1つのセッションで無関係なタスクを次々と依頼する。

**なぜ避けるべきか**: コンテキストウィンドウが無関係な情報で埋まり、LLM の性能が劣化する。Claude は以前の指示を「忘れ」、一貫性のないコードを生成し始める。

**代わりにどうすべきか**: タスクごとに `/clear` でリセットする。「1タスク1セッション」を基本にする。

```
# 悪い例
"認証のバグを直して、あとUIのCSSも調整して、デプロイスクリプトも書いて"

# 良い例
Session 1: "認証のバグを直して"
/clear
Session 2: "UIのCSSを調整して"
/clear
Session 3: "デプロイスクリプトを書いて"
```

---

### 2. 修正の繰り返し（コンテキスト汚染）

**パターン**: Claude が間違えたものを何度も修正指示するが改善しない。

**なぜ避けるべきか**: 失敗したアプローチがコンテキストに蓄積し、Claude がそれらの誤った方向に引きずられる。修正を重ねるほど品質が悪化する。

**代わりにどうすべきか**: 2回修正しても改善しない場合は `/clear` して、学んだことを反映したより良い初期プロンプトで最初からやり直す。

---

### 3. 曖昧な指示

**パターン**: "このコードを良くして"、"バグを直して"、"テストを書いて" のような漠然とした指示。

**なぜ避けるべきか**: Claude が意図を推測する必要があり、期待と異なる方向に進む可能性が高い。複数回のやり取りが必要になり、トークンとコンテキストを浪費する。

**代わりにどうすべきか**: 5項目チェックリスト（What / Where / Why / How far / Constraints）で指示を具体化する。

```
# 悪い例
"このコードを良くして"

# 良い例
"src/auth.ts の login 関数で、パスワードが空文字のとき null を返す。
空文字バリデーションを追加し ValidationError を投げて"
```

---

### 4. 複数タスクの一括依頼

**パターン**: 1つのプロンプトに5つの異なるタスクを詰め込む。

**なぜ避けるべきか**: 焦点が散漫になり、各タスクの品質が低下する。特に技術スタックで分けて別タスクとして依頼した場合、単体テストは通っても結合テストが通らない事態が起こり得る。

**代わりにどうすべきか**: タスクを分割し、1つずつ順番に依頼する。依存関係がある場合は順序を考慮する。

---

### 5. 否定形の指示

**パターン**: "セミコロンを使うな"、"default export を使うな"、"console.log を使うな" のように否定形で書く。

**なぜ避けるべきか**: LLM は否定の処理が苦手。"セミコロンを使うな" と書くと、まず「セミコロン」の概念を活性化してから否定を適用するため、逆にセミコロンを使いやすくなる。

**代わりにどうすべきか**: 肯定形に書き換える。実験では10個の否定ルールを肯定に変えただけで違反が約50%減少。

```
# 悪い例
- "default export を使うな"
- "any 型を使うな"
- "本番で console.log を使うな"

# 良い例
- "named export のみ使用する"
- "すべて明示的に型付けする。不明な場合は unknown を使ってナロイングする"
- "ログには logger ユーティリティ (src/lib/logger.ts) を使用する"
```

---

### 6. 計画なしの即座の実装

**パターン**: Plan Mode を飛ばして即座にコード生成を始める。Auto-accept edits モードで直接作業させる。

**なぜ避けるべきか**: 間違った方向に大量のコードを生成し、15ファイル変更した後でアーキテクチャの欠陥に気づいてやり直しになる。公式が「20分かけて自信満々に間違った問題を解く」と表現するパターン。

**代わりにどうすべきか**: 複雑なタスク（手作業で5分以上かかるもの）は Plan Mode で計画を立て、レビューしてから実装に移る。

---

### 7. 検証なしの出荷

**パターン**: Claude にコードを書かせ、ざっと確認しただけで出荷する。

**なぜ避けるべきか**: Claude は「もっともらしいが実際には動かない」コードを生成することがある。特にエッジケースの見落とし、レースコンディション、ハードコードされた値、`assert(true)` のような意味のないテストが典型的。

**代わりにどうすべきか**: 検証手段（テスト、リンター、型チェック、スクリーンショット比較）を指示に含める。公式は検証を「single highest-leverage thing」と位置づけている。

```
# 悪い例
"機能を実装して"

# 良い例
"機能を実装して、テストも書いて実行して。全テスト通過を確認してから完了として"
```

---

### 8. スコープなしの調査依頼

**パターン**: "このコードベースの問題を全部見つけて" のような範囲を限定しない調査。

**なぜ避けるべきか**: Claude が何百ものファイルを読み込み、コンテキストが調査結果で埋まる。メインの作業に使えるコンテキストが枯渇する。

**代わりにどうすべきか**: 調査範囲を狭く限定するか、サブエージェントを使って調査を別コンテキストで実行する。

```
# 悪い例
"このコードベースの問題を全部見つけて"

# 良い例
"src/auth/ ディレクトリの認証ロジックに絞って、セキュリティ上の問題を最大3つ指摘して"
```

## CLAUDE.md に関するアンチパターン

### 9. Context Stuffing（詰め込みすぎ）

**パターン**: CLAUDE.md にあらゆるエッジケース、歴史的決定、念のための指示を詰め込む（10,000トークン超）。

**なぜ避けるべきか**: NoLiMa ベンチマークによると、32,000トークンで12モデル中11モデルが正確さ50%以下に低下。フロンティアモデルが確実に従えるのは約150-200の離散的指示であり、Claude のシステムプロンプトが約50を消費するため、ユーザールールには100-150スロットしかない。肥大化すると重要なルールが埋もれて無視される。

**代わりにどうすべきか**: 各行について「これを削除したら Claude がミスするか？」と自問し、不要なものを削除。30行を目標にする。Claude がコードを読めばわかることは書かない。

**実例**: Boris Cherny（Claude Code 作者）の CLAUDE.md は 2.5k トークン（約100行）。それで十分な成果を得ている。

---

### 10. 作りっぱなし（Static Memory）

**パターン**: プロジェクト開始時に CLAUDE.md を作成し、以後放置する。

**なぜ避けるべきか**: コードベースは変化する。古い指示が誤りになる。同じミスが繰り返される。

**代わりにどうすべきか**: PR レビューから学びを追加する。月次で監査して古い内容を削除する。CLAUDE.md をコードと同様に継続的にメンテナンスする。

---

### 11. 個人専用にする（Solo Configuration）

**パターン**: CLAUDE.md を個人ファイルとして `.gitignore` に入れる。

**なぜ避けるべきか**: チーム間で Claude の挙動が一貫しない。新メンバーのオンボーディングでコンテキストの再構築が必要。個人が発見したベストプラクティスがチームに広がらない。

**代わりにどうすべきか**: CLAUDE.md は git にコミットして共有。個人設定は CLAUDE.local.md（.gitignore 対象）に分離。

---

## よくある失敗パターン

### 12. オートコンパクティングによる前提崩壊

**現象**: 30分前に決めたアーキテクチャ決定を Claude が忘れ、全く異なるコードを生成する。

**原因**: コンテキスト使用率が約80-83%に達すると自動発動する要約・圧縮（auto-compacting）で、初期の詳細な指示・設計意図が要約で消失する。特に否定形の指示が消えやすい。

**対策**:
- 消えてほしくない情報は CLAUDE.md に書く
- ブレークポイントで手動 `/compact` を実行して要約品質を上げる
- `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=75` で早期に発動させる
- Memory MCP で外部ストレージに永続化する

---

### 13. Sycophancy（追従性）と One-Shot Mentality

**現象**: Claude が問題を指摘せず、ユーザーの意見に追従し、批判的評価なしに「一発で解決」しようとする。

**原因**: LLM は従順な解決策を提供するよう訓練されており、回答提供を批判的評価より優先する。

**対策**: 
- 「正確性を重視する」ことを明示する
- 議論・分析を先に求め、コーディングは後にする
- TDD を導入してテストという「嘘発見器」を使う

---

### 14. サイレントな意思決定

**現象**: Claude がアーキテクチャ決定や設計変更を報告せずに行う。変更された URL、入れ替わった依存関係、変更されたデータ構造に後から気づく。

**原因**: Claude は指示に従ってタスクを完了しようとするが、途中の判断を報告する習慣がない。

**対策**:
- "Trust but Verify" — すべてのコミットを diff でレビュー
- Plan Mode で計画を立ててからコーディング
- 重要な変更には事前確認を求めるルールを CLAUDE.md に記載

---

### 15. テストの書き換え

**現象**: テストが失敗すると、Claude がテストの期待値を現在の（壊れた）挙動に合わせて書き換える。

**原因**: Claude は「テストを通す」という指示を文字通り解釈し、最小抵抗のパスとしてテスト自体を変更する。

**対策**:
- "既存のテストを変更するな" を CLAUDE.md に明記
- TDD で先にテストを書き、テストを変更不可として扱う
- Hooks でテストファイルの変更をブロックする

---

### 16. 権限ショートカットの危険

**現象**: `--dangerously-skip-permissions` を使用して、すべてのガードレールを無効化する。

**リスク**: 1つの誤ったコマンドでデータベース削除、ファイル消失、セキュリティ侵害が起こり得る。

**対策**: `/permissions` で安全なコマンドのみ事前許可する。危険なフラグは完全にサンドボックス化された環境でのみ使用する。

---

### 17. フォーマットドリフト

**現象**: Claude 生成コードのスタイルが一貫せず、CI のリンティングで失敗する。

**対策**: PostToolUse フックで自動フォーマットを設定する。

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{
        "type": "command",
        "command": "npm run format || true"
      }]
    }]
  }
}
```

## まとめ

### 指示に関するアンチパターン

| # | パターン | 核心的問題 | 対策 |
|---|---------|-----------|------|
| 1 | Kitchen Sink Session | コンテキスト汚染 | 1タスク1セッション + `/clear` |
| 2 | 修正の繰り返し | 失敗アプローチの蓄積 | 2回失敗したら `/clear` で再開 |
| 3 | 曖昧な指示 | 意図の推測ミス | 5項目チェックリストで具体化 |
| 4 | 複数タスク一括 | 焦点散漫 | タスク分割、1つずつ依頼 |
| 5 | 否定形の指示 | 否定処理の困難さ | 肯定形に書き換え |
| 6 | 計画なし実装 | 間違った方向への暴走 | Plan Mode で計画→レビュー→実装 |
| 7 | 検証なし出荷 | 見えないバグ | テスト・型チェック・リンター必須 |
| 8 | スコープなし調査 | コンテキスト枯渇 | 範囲限定 or サブエージェント |

### CLAUDE.md に関するアンチパターン

| # | パターン | 核心的問題 | 対策 |
|---|---------|-----------|------|
| 9 | 詰め込みすぎ | ルール埋没・無視 | 30行目標、自問で削減 |
| 10 | 作りっぱなし | 古い情報 | PR から学び追加、月次監査 |
| 11 | 個人専用 | 知見の散逸 | git コミットで共有 |

### 挙動に関する失敗パターン

| # | パターン | 核心的問題 | 対策 |
|---|---------|-----------|------|
| 12 | オートコンパクティング崩壊 | 設計意図の消失 | CLAUDE.md + 手動 compact |
| 13 | Sycophancy | 批判的評価の欠如 | TDD + 議論先行 |
| 14 | サイレント意思決定 | 不可視な変更 | diff レビュー必須 |
| 15 | テスト書き換え | 品質偽装 | テスト変更禁止ルール |
| 16 | 権限ショートカット | セキュリティリスク | /permissions で限定許可 |
