# Phase 1〜3 横断分析レポート

## 概要

Phase 1（指示チャネルの特性: 01〜08）・Phase 2（タスク別指示パターン: 09〜12）・Phase 3（ベストプラクティス・アンチパターン・実例: 13〜15）の全15本の調査成果物を横断的に分析し、共通原則・矛盾・ギャップ・未検証仮説を整理する。

## Step 1 のまとめ: 横断的に浮上するテーマ

全15本を通読した結果、以下のテーマが Phase を横断して繰り返し登場している:

1. **検証手段の提供**: 公式が「single highest-leverage thing」と明言。バグ修正(09)、新機能(10)、リファクタ(11)、レビュー(12)、ベストプラクティス(13) のすべてで最重要とされる

3. **段階的ワークフロー**: Explore → Plan → Code → Commit が 01, 03, 09, 10, 11, 13 で共通パターン
4. **CLAUDE.md の簡潔さ**: 02, 05, 07, 13, 14, 15 で「削ること」の重要性が強調。200行上限、30-100行理想
5. **タスク分割**: 1タスク1セッション、200行以下、30分以内、7ファイル以下が 01, 03, 10, 11, 14 で共通
6. **助言的 vs 決定論的**: CLAUDE.md（助言）、Hooks（決定論的）、settings.json（強制）の3層が 02, 04, 06 で整理
7. **サブエージェントによるコンテキスト分離**: 07, 08, 11, 12 でコンテキスト節約手段として推奨
8. **否定形回避・具体性確保**: 02, 14 で否定形→肯定形の書き換え、01, 11 で具体的指示の優位性

## Step 2: 横断テーマの分析

### 2-1. 共通原則の統合

全 15 本を横断して繰り返し登場する原則を、重要度順にまとめる。

#### 原則 1: 検証手段の提供が最高レバレッジ（全 Phase で最重要）

公式ドキュメントが「single highest-leverage thing」と明言。Phase 2 の全タスク種別（09〜12）および Phase 3 のベストプラクティス（13）で一貫して最重要とされる。

- **バグ修正（09）**: TDD Bug Fix パターン — 失敗テスト作成 → 修正 → テスト通過確認
- **新機能（10）**: 仕様にテストケース・期待値を事前記載し、実装前に検証基準を確定
- **リファクタリング（11）**: Characterization Test で振る舞いを固定してから変更
- **コードレビュー（12）**: 信頼度スコア（80+）で偽陽性を除去、Hooks で lint/型チェック自動実行
- **ベストプラクティス（13）**: テスト・スクリーンショット・期待出力を事前に渡す

**統合原則**: 検証手段は「実装後に確認するもの」ではなく「実装前に与えるもの」。Claude にセルフチェック能力を付与するという設計思想が一貫している。

#### 原則 2: コンテキスト管理が全設計原則の基盤

08 で専門的に扱われるが、01, 02, 05, 07, 10, 11, 13 でも繰り返し言及される。「ほとんどのベストプラクティスはコンテキストウィンドウが急速に埋まるという制約から導かれる」（01）。

- **閾値**: 20-40% で品質低下開始、60% で手動 `/compact` 推奨、95% で自動圧縮（品質最悪）
- **手段**: `/clear`（タスク切替）、`/compact`（圧縮）、`/btw`（履歴非汚染の質問）、サブエージェント委譲
- **設計への影響**: CLAUDE.md の簡潔さ、タスク分割、セッション管理のすべてがこの制約から導かれる

#### 原則 3: 段階的ワークフロー（Explore → Plan → Code → Commit）

01, 03, 09, 10, 11, 13 で共通パターンとして登場。

- 複雑なタスクは Plan mode で計画 → 実装 → 検証 → コミットの 4 段階
- 「差分を一文で説明できるか」テストで Plan mode の要否を判断（03）
- Plan mode で初回成功率が 37% 向上（10、ただしブログデータで再現性は不明）
- 単純タスク（5分以内、1-2ファイル）では計画オーバーヘッドを避ける

#### 原則 4: タスク分割と 1 タスク 1 セッション

01, 03, 10, 11, 14 で共通する閾値:

| 指標 | 推奨値 | 出典 |
|------|--------|------|
| コード変更量 | 200 行以下/タスク | コミュニティ |
| 同時変更ファイル | 7 ファイル以下 | コミュニティ（DataCamp） |
| 所要時間 | 30 分以内/タスク | コミュニティ |
| セッション | 1 タスク = 1 セッション | 公式ドキュメント |

#### 原則 5: CLAUDE.md は簡潔に保ち、段階的に開示する

02, 05, 07, 13, 14, 15 で「削ること」の重要性が強調:

- 30〜100 行が理想、200 行が上限（02, 15）
- 「この行を削除したら Claude がミスするか？」テスト（02）
- Progressive Disclosure: CLAUDE.md（概要）→ `.claude/rules/`（トピック別）→ Skills（オンデマンド）
- コードから自明なことは書かない（07, 13）

#### 原則 6: 助言的 vs 決定論的の 3 層構造

02, 04, 06 で整理される強制力の段階:

| レイヤー | 強制力 | 用途 |
|----------|--------|------|
| CLAUDE.md | 助言的（advisory） | コーディング規約、プロジェクト知識 |
| Hooks | 決定論的（deterministic） | lint 自動実行、ファイル保護 |
| settings.json | 強制（enforced） | 許可ツール制限、パーミッション |

### 2-2. 矛盾・ギャップの特定

#### 矛盾・テンション

**① CLAUDE.md の行数推奨が文書間で揺れている**

- 02: 200 行以下/ファイル
- 15: 実例は 30〜120 行。Anthropic 自身のプロジェクトは約 60 行
- 14: 成功例は約 100 行（2.5K トークン）

→ **解決案**: 30〜100 行が実用的理想値、200 行は絶対上限と整理できる。矛盾ではなく粒度の違い。

**② 具体的指示 vs 曖昧な探索指示の価値**

- 01: 「曖昧なプロンプトは探索・学習時に価値がある」
- 10: 「指示が具体的であるほど修正回数が減る」
- 11: 「抽象的指示（『いい感じにして』）は振る舞い変更リスクが高い」

→ **解決案**: タスクモードで使い分け。ビルド/デバッグ → 具体的、ラーニング/探索 → 曖昧でも可。矛盾ではなくコンテキスト依存。

**③ Plan mode の必要性判断が文書間で異なる**

- 03: 「差分を一文で説明できるなら Plan 不要」
- 09（バグ修正）: Plan mode にほとんど触れず、TDD パターンに直行
- 11（リファクタリング）: Plan mode を重要視

→ **解決案**: タスク種別で分岐。バグ修正は TDD 駆動（Plan は調査が複雑な場合のみ）、リファクタリング・新機能は原則 Plan 使用。

**④ セッション分割の閾値が不明確**

- 01: `/clear` をタスク間で使う
- 14: Kitchen Sink セッションはアンチパターン
- しかし「どの時点でセッションを分割すべきか」の明確な判断基準がない

→ **ギャップ**: 「60% ルール」「2 回修正失敗」以外のセッション分割トリガーが未定義。

#### カバーされていないギャップ

| ギャップ | 説明 | 重要度 |
|---------|------|--------|
| **チームワークフロー** | 複数人が同一ブランチで Claude Code を使う場面の指針がない | 高 |
| **CI/CD 統合** | `-p` フラグの言及のみ。パイプライン例・自動化レシピがない | 高 |
| **多言語プロジェクト** | TypeScript + Python + Go 混在時の CLAUDE.md 戦略が未定義 | 中 |
| **コスト最適化** | Opus/Sonnet/Haiku の使い分け、Fast mode の判断基準がない | 中 |
| **段階的導入** | 既存プロジェクトへのレトロフィット手順がない | 中 |
| **セキュリティモデル** | permissions.deny 以外の包括的セキュリティフレームワークがない | 中 |
| **失敗モード** | フレーキーテスト、レースコンディション、E2E テスト戦略が未対応 | 低 |

### 2-3. 「コンテキスト管理」の横断整理

コンテキスト管理を軸に、各タスクパターンでの戦略を統合する。

#### トークン消費の実態

| 項目 | トークン数 | 出典 |
|------|-----------|------|
| システムプロンプト | ≈4,200 | 07 |
| CLAUDE.md（プロジェクト） | ≈1,800 | 07 |
| オートメモリ（MEMORY.md） | ≈680 | 07 |
| **セッション起動時の固定コスト** | **≈7,850** | 07 |
| ファイル 1 本の読み込み（例: src/auth.ts） | ≈2,400 | 07 |
| Grep 結果 | ≈600 | 07 |
| サブエージェント探索 → 要約 | 6,100 → 420 | 07 |

#### タスク規模別コンテキスト戦略

| 規模 | ファイル数 | 推奨戦略 |
|------|-----------|---------|
| 小（5分以内） | 1-2 | 直接指示。Plan 不要。セッション内で完結 |
| 中（30分以内） | 3-5 | Plan mode で計画 → `/compact` → 実装。1セッション |
| 大（30分超） | 6+ | 仕様策定セッション + 実装セッション（各200行以下）に分割 |
| 大規模移行 | 多数 | Strangler Fig + バッチスクリプト。セッション間は仕様文書で引き継ぎ |

#### コンテキスト保全の 3 つの武器

1. **`/compact`（60% で発動）**: 「何を残すか」を明示指定（例: `/compact preserve API changes and test results`）
2. **`/clear`（タスク切替時）**: CLAUDE.md が再注入される唯一の完全リセット手段
3. **サブエージェント委譲**: 探索をサブエージェントに委譲し、要約のみ受け取る（127倍の圧縮率）

#### CLAUDE.md はコンテキストの「アンカー」

- 圧縮を生き残る唯一の永続指示（08）
- セッション起動時に毎回再注入される（06, 08）
- だからこそ簡潔でなければならない（肥大化 = 全セッションのベースラインコスト増）

### 2-4. CLAUDE.md 関連知見の統合

Phase 1-2 の活用法、Phase 3 の実例、各 Phase で言及される CLAUDE.md の役割を統合する。

#### CLAUDE.md の統合的位置づけ

```
CLAUDE.md の役割マップ
├── 永続的な指示の唯一の保証手段（08: 圧縮を生き残る）
├── セッション開始時のコンテキストコスト（07: ≈1,800 トークン）
├── ユーザーメッセージとして注入（02: システムプロンプトではない）
├── 助言的（advisory）— 強制力はない（04, 06）
└── Progressive Disclosure の入口（05, 07, 15）
    ├── Layer 1: CLAUDE.md — 高シグナルのみ
    ├── Layer 2: .claude/rules/ — トピック別、パス条件付き
    └── Layer 3: .claude/skills/ — オンデマンド起動
```

#### 書くべきこと vs 書くべきでないこと

| 書くべき（13, 02） | 書くべきでない（07, 14） |
|-------------------|----------------------|
| Claude が推測できないコマンド（例: `bun test`） | コードから自明なこと |
| 非デフォルトのコーディングスタイル | 標準的な構文規則 |
| テスト手順・検証コマンド | 長い API ドキュメント（リンクで代替） |
| プロジェクト固有のアーキテクチャ | ファイル個別の説明 |
| 「Things That Will Bite You」（罠） | 頻繁に変わる情報 |

#### 記述スタイルの推奨

- **事実形式 > 命令形式**: 「Bun を使っている」 > 「Bun を使え」（02）
- **具体例を添える**: 「2-space indent」 > 「適切にフォーマットする」（13）
- **否定形 → 肯定形**: 「〜するな」→「代わりに〜する」で違反率 ≈50% 減（14）
- **削除テスト**: 各行について「これを消したら Claude がミスするか？」を問う（02）

#### 実例からのベストプラクティス（15）

| プロジェクト | 行数 | 特徴 |
|------------|------|------|
| Anthropic claude-code-action | ≈60 | 「Things That Will Bite You」セクション |
| tldraw/make-real | ≈120 | 「Code Patterns」で新機能追加手順をステップ記載 |
| ShopFront（Zenn 例） | ≈30 | モーダル構成、`@docs/auth-flow.md` で詳細分離 |
| Boris Cherny | ≈100 | Git コミット、PR レビューからの学習ループ |

#### テンション: 簡潔さ vs 具体性

- 07, 14: CLAUDE.md は短いほど良い（肥大化で重要ルールが埋没）
- 02, 13: 具体的であるべき（曖昧な指示は無視される）
- **解決**: 具体的だが簡潔に。1 行で具体例まで含む記述（例: `"2-space indent, no tabs"`）

### 2-5. 「検証手段の提供」の横断整理

Phase 3 で最重要とされた「検証手段を与える」が、各タスク種別でどう適用されるかを整理する。

#### タスク種別ごとの検証パターン

| タスク種別 | 一次検証手段 | 二次検証手段 | タイミング |
|-----------|------------|------------|-----------|
| バグ修正（09） | 失敗テスト → 修正 → パス（TDD） | 回帰テスト全実行 | 修正前にテスト作成 |
| 新機能（10） | 仕様内のテストケース・期待値 | `npm run build && npm run lint` | 仕様策定時に定義 |
| リファクタリング（11） | Characterization Test（振る舞い固定） | lint + 型チェック | リファクタ前に作成 |
| コードレビュー（12） | 信頼度スコア（80+） | Hooks による lint/型チェック自動実行 | レビュー実行時 |

#### 統合原則: 検証は「事前に与える」

全タスク種別で共通するのは、**検証手段を実装前に提供する**という設計思想:

1. **バグ修正**: まず再現テストを書き、「このテストが通れば修正完了」と定義
2. **新機能**: 仕様書にテストケース（例: `user@example.com → true`）を含め、実装前に期待値を確定
3. **リファクタリング**: 既存の振る舞いを Characterization Test で固定し、「テストが通り続ければ正しい」と定義
4. **コードレビュー**: レビュー基準・重大度定義を事前設定し、信頼度閾値で偽陽性を制御

#### 検証コマンドの CLAUDE.md への記載

13 のベストプラクティスに従い、検証コマンドは CLAUDE.md に記載すべき最優先項目:

```markdown
## テスト・検証
- テスト: `npm test`
- 型チェック: `npx tsc --noEmit`
- lint: `npm run lint`
- ビルド確認: `npm run build`
```

これにより Claude が自律的に検証ループを回せるようになる。

#### 検証レベルの段階

```
Level 0: 検証手段なし（最もリスクが高い — アンチパターン）
Level 1: テストコマンドを CLAUDE.md に記載（最低限）
Level 2: 各タスクに具体的なテストケース・期待値を提供
Level 3: Hooks で lint/型チェックを自動実行（決定論的保証）
Level 4: Writer/Reviewer パターンで別セッションから検証（バイアス排除）
```

---

### Step 2 のまとめ

横断分析から得られた主要な洞察:

1. **最重要原則は 2 つ**: 「検証手段の事前提供」と「コンテキスト管理」。この 2 つが他のすべてのプラクティスの土台
2. **矛盾は少ない**: 文書間の「矛盾」は多くの場合コンテキスト依存の使い分け（具体的 vs 曖昧、Plan 要否など）
3. **最大のギャップはチーム・CI/CD**: 全調査が単一ユーザー前提。チームワークフローと CI/CD 統合が未開拓
4. **CLAUDE.md は「簡潔な具体性」が鍵**: 短くかつ具体的、Progressive Disclosure で詳細を分離
5. **閾値は目安として有用だが未検証**: 200行、7ファイル、30分、60% など数値基準が多いが、定量的検証はほぼない

## Step 4: 追加調査の方向性決定

Step 3 で分類した 3 カテゴリ（実験 15 件・追加調査 9 件・ガイドライン化 8 件）の投資対効果を評価し、Phase 4 以降の計画を策定する。

### 4-1. 優先順位付けの方針

#### 評価軸

| 軸 | 説明 |
|----|------|
| **即効性** | 成果物を出すまでの所要コストが低いか |
| **実用インパクト** | 日常ワークフローの改善に直結するか |
| **知見の成熟度** | すでに十分な情報があるか、追加調査が必要か |
| **検証可能性** | 実験として再現可能な設計ができるか |

#### 判断: 3 フェーズに分割して段階的に進める

```
Phase 4A: ガイドライン化（低コスト・確実な成果）
    ↓ 成果物を使いながら
Phase 4B: 実証実験（コア仮説の検証）
    ↓ 実験結果をガイドラインにフィードバック
Phase 4C: 追加 Web 調査（ギャップ埋め）
```

理由:
- Phase 4A は既存知見のまとめであり、追加調査不要で着手できる
- Phase 4B は 4A のガイドラインを「使いながら」検証する形になり、実験設計が自然に定まる
- Phase 4C は 4A・4B の結果を踏まえて調査対象を絞り込める（無駄な調査を避けられる）

---

### 4-2. Phase 4A: ガイドライン化（知見の実用化）

**目的**: Phase 1〜3 で十分な裏付けがある知見を、即座に使える成果物に変換する。

#### 対象と成果物

| # | テーマ | 成果物 | 優先度 | 根拠 |
|---|--------|--------|--------|------|
| A1 | CLAUDE.md テンプレート | `templates/claude-md-template.md` | 最高 | 全 Phase の知見が集約する中心成果物。WHAT/WHY/HOW フレームワーク + 検証コマンド + Things That Will Bite You。30〜100 行、事実形式、肯定形 |
| A2 | タスク別プロンプトテンプレート | `templates/task-prompts/{bug-fix,new-feature,refactoring,code-review}.md` | 高 | Phase 2 の 4 タスク種別を即座に使える雛形に変換 |
| A3 | 段階的ワークフローガイド | `guides/workflow.md` | 高 | Explore → Plan → Code → Commit の 4 フェーズ + タスク規模別判断基準 |
| A4 | コンテキスト管理チートシート | `guides/context-management.md` | 中 | トークン消費目安、60% ルール、/compact・/clear・サブエージェントの使い分け |
| A5 | 検証手段の設計ガイド | `guides/verification.md` | 中 | タスク種別ごとの検証パターン + 検証レベル（Level 0〜4） |

#### 進め方

- A1（CLAUDE.md テンプレート）を最初に作成し、このリポジトリ自身の CLAUDE.md を改善する実践例とする
- A2〜A5 は並行して作成可能
- 各ガイドラインは「Phase 1〜3 の調査に基づく推奨」と明記し、Phase 4B の実験で検証予定であることを注記する

### 4-3. Phase 4B: 実証実験（コア仮説の検証）

**目的**: 調査ベースの知見のうち、定量的な裏付けがない重要仮説を実験で検証する。

#### 実験の選定基準

Step 3-1 の 15 件から、以下の基準で絞り込む:
- 日常ワークフローへの影響度が高い
- 実験設計が比較的シンプル（変数を制御しやすい）
- Phase 4A のガイドラインの根拠を強化する

#### 選定した実験（6 件）

| # | 仮説 | 実験概要 | 出典 | 選定理由 |
|---|------|---------|------|---------|
| B1 | 事実形式は命令形式より遵守率が高い | 同一タスクを事実形式/命令形式の CLAUDE.md で複数回実行し遵守率を比較 | 02 | A1 テンプレートの記述スタイル根拠 |
| B2 | 否定形→肯定形で違反率 ≈50% 減少 | 否定形/肯定形ルールで同一タスクを反復実行し違反率を集計 | 14 | A1 テンプレートの記述ルール根拠 |
| B3 | CLAUDE.md 行数と遵守率の関係 | 30行/100行/200行超の CLAUDE.md で遵守率を比較 | 02, 14, 15 | A1 テンプレートの行数ガイドライン根拠 |
| B4 | Plan mode の初回成功率向上 | 中規模タスク（3-5 ファイル変更）を Plan あり/なしで複数回実行 | 03, 10 | A3 ワークフローガイドの根拠 |
| B5 | TDD Bug Fix パターンの効果 | 既知バグに対し TDD パターン/直接修正の成功率・ターン数を比較 | 09 | A2 バグ修正テンプレートの根拠 |
| B6 | Hooks（lint 自動実行）のコード品質への影響 | Hooks 有/無で同一タスクを実行し lint エラー数・テスト通過率を比較 | 04 | A5 検証手段ガイドの根拠 |

#### 除外した実験と理由

| 仮説 | 除外理由 |
|------|---------|
| コンテキスト 60% 閾値の品質低下 | 意図的にコンテキストを制御する実験設計が困難 |
| 7 ファイル超の品質低下 | 大量のタスクバリエーションが必要でコスト高 |
| 思考キーワードの効果 | 優先度低（日常的に使う場面が限定的） |
| Skills の自動起動精度 | ツール機能テストであり、ワークフロー改善への直接貢献が低い |
| オートメモリの効果 | マルチセッション実験の設計が複雑 |
| /compact preserve の情報保持率 | 測定基準の定義が困難 |
| サブエージェント vs 直接探索 | 中優先度で Phase 4C 後に再検討 |
| Writer/Reviewer パターン | 中優先度で Phase 4C 後に再検討 |
| Things That Will Bite You セクション | B1〜B3 の結果から推定可能 |

#### 実験設計の共通方針

- **制御変数**: 1 実験につき 1 変数のみ変更（例: CLAUDE.md の記述スタイルだけ変える）
- **反復回数**: 各条件 5〜10 回（統計的に有意な差を検出するため）
- **評価指標**: 遵守率、成功率、所要ターン数、エラー数など定量的指標を使用
- **対象プロジェクト**: 小規模なサンプルプロジェクトを用意し、実験間で共有する
- **コスト管理**: Sonnet を主に使用し、Opus は B4（Plan mode 効果）の比較実験のみ

### 4-4. Phase 4C: 追加 Web 調査（ギャップ埋め）

**目的**: Phase 1〜3 でカバーされなかった領域のうち、Phase 4A・4B の成果を拡張するために必要な情報を収集する。

#### 選定した調査テーマ（3 件）

| # | テーマ | 調査の方向性 | 選定理由 |
|---|--------|-------------|---------|
| C1 | CI/CD 統合パターン | claude-code-action の最新ドキュメント、GitHub Actions 連携事例、ヘッドレスモードの実践レシピ | Phase 4A のガイドラインを CI/CD に拡張するために不可欠 |
| C2 | チームワークフロー | 複数人での CLAUDE.md 管理、コンフリクト回避、レビュー分担の事例 | 個人利用→チーム展開のステップに必要 |
| C3 | コスト最適化 | Opus/Sonnet/Haiku の使い分け基準、トークン消費モニタリング | Phase 4B の実験設計にも影響する実用的な情報 |

#### 後回しにした調査テーマと理由

| テーマ | 理由 |
|--------|------|
| 多言語プロジェクト | 当リポジトリが単一言語寄りで即効性が低い |
| 段階的導入 | Phase 4A のガイドラインが導入ガイドの役割を果たす |
| セキュリティモデル | 個人利用フェーズでは優先度低 |
| 多エージェント合議制 | Phase 4B の実験結果を見てから判断 |
| 失敗モード対策 | 特定状況向けで汎用性が低い |
| 大規模コードベース | 当リポジトリの規模では検証困難 |

- Phase 4A のガイドラインは Phase 4B の実験結果に基づいて改訂する

- Phase 4C の調査結果は Phase 4A のガイドラインに反映する
