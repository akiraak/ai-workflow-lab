# B3: CLAUDE.md 行数と遵守率の関係

## 仮説

CLAUDE.md の行数が増えるほど、コアルールの遵守率が低下する。
30 行と 200 行超では有意な遵守率差が生じる。

- 元調査: [02-claude-md.md](02-claude-md.md)（200 行上限）、[14-anti-patterns.md](14-anti-patterns.md)（詰め込みアンチパターン）、[15-claude-md-examples.md](15-claude-md-examples.md)（30-100 行が理想）
- 検証対象: [A1 CLAUDE.md テンプレート](../../templates/claude-md-template.md) の行数ガイドライン

---

## 実験デザイン

### 独立変数（操作する変数）

CLAUDE.md の行数（3 条件）:

| 条件 | 行数 | 内容 |
|------|------|------|
| A: 短い | 30 行 | コアルール 10 個 + 開発コマンドのみ |
| B: 中程度 | 100 行 | 条件 A + アーキテクチャ説明 + Things That Will Bite You + 具体例 |
| C: 長い | 200 行超 | 条件 B + API ドキュメント + ファイル個別説明 + 冗長な補足 |

**3 条件すべてにコアルール 10 個（B1 と同一）を含める。** 条件 B/C は追加情報がコアルールの遵守を邪魔するかを検証する。

### 従属変数

- **コア 10 ルールの遵守率**: 遵守数 / 10
- 条件 B/C の追加情報の遵守率は測定しない（変数が増えすぎるため）

### 統制変数

| 項目 | 値 |
|------|-----|
| モデル | claude-sonnet-4-20250514 |
| タスクプロンプト | B1 と同一 |
| 反復回数 | 各条件 7 回（合計 21 回） |
| ルール記述スタイル | 事実形式で統一（B1 の条件 A と同じ） |

## CLAUDE.md バリアント

### 条件 A: 30 行（コアのみ）

```markdown
# Sample Project

TypeScript のショッピングカートアプリケーション。

## 開発コマンド

- テスト: `npx vitest run`
- 型チェック: `npx tsc --noEmit`
- lint: `npx eslint src/ tests/`

## コードスタイル

- named export を使用している
- エラー処理は例外ではなく Result 型で返している
- テストファイルは tests/ ディレクトリに配置している
- 関数名は camelCase を採用している
- 関数の引数と戻り値に明示的な型アノテーションを付けている
- 1 関数は 30 行以下で書いている
- 公開関数には JSDoc コメントを付けている
- テストは describe/it で構造化している
- import は相対パスではなく @/ エイリアスを使用している
- ログ出力には logger ユーティリティを使用している
```

### 条件 B: 100 行（コア + 補足情報）

条件 A の全内容に加え:
- アーキテクチャ概要（src/ ディレクトリの各ファイルの役割）
- Things That Will Bite You（3-5 項目の注意事項）
- 具体的なコード例（Result 型の使い方、テストの書き方）
- Git コミット規約

### 条件 C: 200 行超（詰め込みすぎアンチパターン）

条件 B の全内容に加え:
- 各ファイルの詳細な説明（cart.ts, discount.ts, validator.ts, ...）
- API エンドポイントの説明（実際には存在しないが CLAUDE.md に記載）
- 長い歴史的経緯の説明
- 冗長なスタイルガイド（すでに ESLint でカバーされている内容を繰り返す）
- 個別関数の使い方ドキュメント
