# B2: 否定形 vs 肯定形の違反率比較

## 仮説

CLAUDE.md のルールを肯定形（「〜する」）で書くと、否定形（「〜しない」「〜するな」）で書いた場合と比べて違反率が約 50% 減少する。

- 元調査: [14-anti-patterns.md](14-anti-patterns.md) — 「否定形→肯定形の書き換えで違反が約 50% 減少」（コミュニティ実験報告）
- 検証対象: [A1 CLAUDE.md テンプレート](../../templates/claude-md-template.md) のルール表現推奨

---

## 実験デザイン

### 独立変数（操作する変数）

CLAUDE.md のルール表現（2 条件）:

| 条件 | スタイル | 例 |
|------|---------|-----|
| A: 肯定形 | 「〜する」「〜を使用する」 | 「named export のみ使用する」 |
| B: 否定形 | 「〜しない」「〜を使わない」 | 「default export を使わない」 |

同一の 10 ルールの意味を肯定/否定で表現。

### 従属変数（測定する指標）

- **違反率**: 違反ルール数 / 10（各試行ごと）
- **各ルールの遵守/違反**: バイナリ判定

### 統制変数

| 項目 | 値 |
|------|-----|
| モデル | claude-sonnet-4-20250514 |
| タスクプロンプト | B1 と同一 |
| サンプルプロジェクト | `experiments/phase-4b/sample-project/` |
| 反復回数 | 各条件 10 回（合計 20 回） |

## 10 ルールセット

| # | ルール内容 | 肯定形 (条件A) | 否定形 (条件B) |
|---|-----------|---------------|---------------|
| 1 | named export | 「named export のみ使用する」 | 「default export を使わない」 |
| 2 | Result パターン | 「エラーは Result 型で返す」 | 「例外を throw しない」 |
| 3 | テスト配置 | 「テストは tests/ に配置する」 | 「テストを src/ 内に置かない」 |
| 4 | camelCase | 「関数名は camelCase で書く」 | 「関数名に snake_case や PascalCase を使わない」 |
| 5 | 型アノテーション | 「引数と戻り値に型を明示する」 | 「型アノテーションを省略しない」 |
| 6 | 関数サイズ | 「1 関数 30 行以下に収める」 | 「1 関数が 30 行を超えないようにする」 |
| 7 | JSDoc | 「公開関数に JSDoc を付ける」 | 「公開関数の JSDoc を省略しない」 |
| 8 | テスト構造 | 「describe/it でテストを構造化する」 | 「describe/it を使わずにテストを書かない」 |
| 9 | パスエイリアス | 「import は @/ エイリアスを使用する」 | 「import に相対パス（./）を使わない」 |
| 10 | ロガー | 「ログは logger ユーティリティで出力する」 | 「console.log を使わない」 |

## CLAUDE.md テンプレート

### 条件 A: 肯定形

```markdown
# Sample Project

TypeScript のショッピングカートアプリケーション。

## 開発コマンド

- テスト: `npx vitest run`
- 型チェック: `npx tsc --noEmit`
- lint: `npx eslint src/ tests/`

## コードスタイル

- named export のみ使用する
- エラーは Result 型（`{ success: boolean; data?: T; error?: string }`）で返す
- テストは tests/ ディレクトリに配置する
- 関数名は camelCase で書く
- 関数の引数と戻り値に型を明示する
- 1 関数は 30 行以下に収める
- 公開関数に JSDoc コメントを付ける
- テストは describe/it で構造化する
- import は @/ エイリアスを使用する
- ログは logger ユーティリティで出力する（src/logger.ts の logger を使う）
```

### 条件 B: 否定形

```markdown
# Sample Project

TypeScript のショッピングカートアプリケーション。

## 開発コマンド

- テスト: `npx vitest run`
- 型チェック: `npx tsc --noEmit`
- lint: `npx eslint src/ tests/`

## コードスタイル

- default export を使わない
- 例外を throw しない。Result 型（`{ success: boolean; data?: T; error?: string }`）で返す
- テストを src/ 内に置かない。tests/ ディレクトリに配置する
- 関数名に snake_case や PascalCase を使わない
- 型アノテーションを省略しない
- 1 関数が 30 行を超えないようにする
- 公開関数の JSDoc コメントを省略しない
- describe/it を使わずにテストを書かない
- import に相対パス（./）を使わない
- console.log を使わない。logger を使う（src/logger.ts の logger を使う）
```

## 判定基準

B1 と同一の自動判定スクリプトを使用。

否定形の条件 B で「やるなと言われたことをやった」ケースを違反としてカウント。

## 予想される結果

- 仮説支持: 否定形（条件B）の違反率が肯定形（条件A）の約 2 倍（例: 肯定形 15% vs 否定形 30%）
- 仮説不支持: 違反率の差が 10% 未満
