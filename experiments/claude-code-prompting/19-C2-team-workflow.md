## 1. チームでの CLAUDE.md 管理

### 階層構造による管理

CLAUDE.md は3つのレベルで階層的に管理するのがベストプラクティス。

| レベル | ファイル | git | 用途 |
|--------|---------|-----|------|
| 個人 | `~/.claude/CLAUDE.md` | — | 個人のコーディング規約、好みの設定 |
| プロジェクト | `./CLAUDE.md` | チェックイン | 禁止パターン、必須ツール、認証標準 |
| ローカルオーバーライド | `./CLAUDE.local.md` | `.gitignore` | 個人用のプロジェクト固有メモ |

### ゴールデンルール

Claude Code 創作者の Boris Cherny（Anthropic Staff Engineer）が提唱する原則:

> **「Claude が間違ったことをするたびに CLAUDE.md に追記して、次回は繰り返さないようにする」**

CLAUDE.md は git にチェックインし、チーム全体で共有し、週に何度も更新する「生きたドキュメント」として運用する。

### コンテンツの原則

- **1ページ以内** に収める（DoorDash Hannah Stulberg の推奨）。長すぎると 80% のセッションで不要な情報にコンテキストを消費
- **200行以下** を目標にし、必要なスキルは import 文でリンク（Ran Isenberg の実践）
- 定期的にプルーニングし、Claude が正しく動作している事項の記述は削除する

### チーム規模別の最適化

| チーム規模 | 推奨アプローチ |
|-----------|--------------|
| 2-5人 | シンプルな統一設定ファイル |
| 5-15人 | フロントエンド/バックエンド等のロール別に分離 |
| 15人以上 | 専任の AI ワークフローエンジニア、ドキュメントメンテナー配置 |

### Team OS パターン（DoorDash PM 提唱）

共有 GitHub リポジトリに以下を構成:
- Root CLAUDE.md にドキュメントインデックス、チーム名簿、Slack チャンネル一覧
- 各フォルダに Nested Doc Index（ナビゲーションマップ）
- トークン効率の3層構造: Tier 1（常時ロード ~500 tokens）→ Tier 2（クエリ時ロード）→ Tier 3（オンデマンドロード）

## 2. コンフリクト回避

### Git Worktrees による分離（最重要手法）

複数の Claude Code セッションが同時に動く際の **最も推奨される解決策は Git Worktrees**。

```bash
claude --worktree feature-auth   # Terminal 1
claude --worktree bugfix-123     # Terminal 2
claude --worktree experiment     # Terminal 3
```

- Claude Code v2.1.50 以降で `--worktree` フラグを直接サポート
- 各セッションが独立したチェックアウト・ブランチ・インデックスを持つ
- Boris Cherny は **10-15 の Claude セッションを同時実行**し、全て別々の git worktree を使用（チームの「#1 生産性ティップ」）

### Agent Teams でのファイル分離

Agent Teams 使用時の鉄則: **複数のエージェントに同じファイルを触らせない**。

- ドメイン（フロントエンド/バックエンド/テスト等）でファイル所有権を明確に分離
- タスクの claim にはファイルロッキングが使用され、レース条件を防止

### 使い分けの判断基準

| 並列化すべき場面 | 単一セッションの方が良い場面 |
|-----------------|--------------------------|
| リサーチ・レビュー | 順序依存タスク |
| 独立した新モジュール開発 | 同一ファイル編集 |
| 競合仮説でのデバッグ | 依存関係の多い作業 |
| クロスレイヤー変更 | |

## 3. レビュー分担

### Claude Code Review（公式マネージドサービス）

Team/Enterprise プランで利用可能な自動 PR レビュー機能。

| 特徴 | 説明 |
|------|------|
| 並列分析 | 専門エージェント群がロジックエラー、セキュリティ脆弱性、エッジケース、リグレッションを分析 |
| 重要度タグ | 🔴 Important（マージ前修正必須）、🟡 Nit（軽微）、🟣 Pre-existing（既存バグ） |
| 既存フロー共存 | PR を承認もブロックもしない |
| カスタマイズ | `REVIEW.md` ファイルでレビュー方針を指定 |

### 人間と AI のレビュー分担

| 担当 | 責務 |
|------|------|
| **AI（Claude）** | バグ検出、セキュリティ脆弱性、テストカバレッジ確認、コーディング規約違反の指摘（ファーストラウンド） |
| **人間** | アーキテクチャ設計判断、ビジネスロジックの妥当性、長期的保守性、チーム固有のコンテキスト |

### AI 生成コードの4つのレビュー観点

1. **プロンプト品質**: リクエストは十分に具体的だったか
2. **アーキテクチャ適合**: 既存システムと矛盾しないか
3. **セキュリティ**: 認証フロー、クエリ、エンドポイントは人間レビュー必須
4. **保守性**: コードは理解可能でスケーラブルか

### Writer/Reviewer パターン

- Session A（Writer）: 実装を担当
- Session B（Reviewer）: 別の新鮮なコンテキストでレビュー（自分が書いたコードに対するバイアスがない）

## 4. チーム導入事例

### 成功事例

| 企業 | 成果 |
|------|------|
| **Altana**（AI サプライチェーン） | エンジニアリングチーム全体で開発速度 2-10倍 |
| **TELUS**（テレコム大手） | 57,000人がアクセス、月間1,000億トークン以上、コード配信速度30%向上 |
| **Behavox** | 数百人の開発者に展開、「頼りになるペアプログラマー」として定着 |
| **Novo Nordisk** | 分子生物学博士号を持つ非エンジニアがプロトタイプ開発に参加 |
| **Epic** | 利用の半分以上が非開発者ロール（サポート・導入スタッフ） |
| **Rakuten** | 7時間の自律コーディングで数週間の作業を完了 |

### 失敗パターン

- Claude Code を「魔法のソリューション」として扱い、曖昧な要件を投げて洗練されたコードを期待する
- AI 生成コードのガバナンス（レビュー、バージョン管理、テスト）を確立しないまま導入
- 各開発者が独自のプロンプティングスタイルを使い、一貫性のないコードが生成される
- 初日で諦める（効果的な Team OS の構築には数週間かかる）

### 導入の教訓

- **ドメイン専門知識がボトルネック**: ツールの能力より、開発者の専門知識と適切なコンテキスト提供が成功の鍵
- **文化的変革が必要**: 技術設定だけでなく、AI をチームメンバーとして扱う文化を育てる
- **ROI の発現時期**: 運用自動化で初月から効果あり

## 5. ガバナンス・標準化

### managed-settings.json による組織ポリシー強制

Enterprise プランで利用可能。開発者がオーバーライドできないポリシーを強制:

- `disableBypassPermissionsMode` で `--dangerously-skip-permissions` を無効化
- `.env`、シークレット、SSH キーの読み取り禁止
- MCP サーバーのホワイトリスト管理（`allowManagedMcpServersOnly: true`）
- 配信: サーバー管理 or MDM（Jamf/Kandji/Intune）経由

### 品質ゲートの3層構造

| レイヤー | メカニズム | 機能 |
|----------|-----------|------|
| 予防 | CLAUDE.md 制約 | 禁止パターンの文書化 |
| 実行時 | Hooks バリデーション | yamllint、pytest の自動実行 |
| 事後検証 | GitHub Actions | ビルド成功、デプロイチェック |

### 3レーンスケーリングフレームワーク（Turing 提唱）

| レーン | 内容 | リスクレベル |
|--------|------|------------|
| Lane A（ローカル探索） | サンドボックスでの低リスクなコード分析・単体テスト | 低 |
| Lane B（CI 連携変更） | 全プロダクション変更が CI/CD パイプラインを通過 | 中 |
| Lane C（リリース・デプロイ） | 人間が最終リリース権限を保持 | 高 |

### 段階的展開

| フェーズ | 規模 | 期間 | 内容 |
|---------|------|------|------|
| Phase 1（パイロット） | 5-10人 | 2週間 | managed settings, MCP ホワイトリスト, 支出上限テスト |
| Phase 2（部門展開） | 20-50人 | — | リポジトリと MCP 統合を拡大 |
| Phase 3（全社展開） | 全社 | — | MDM で managed settings を全フリートに配布 |

### チーム標準化の具体策

- **共有プロンプティングプレイブック**: 全メンバーが同じスタイルで Claude にリクエスト
- **共有カスタムコマンド** (`.claude/commands/`): feature-start, code-review, hotfix, release 等を標準化
- **環境別権限ポリシー**: 開発（フル権限）、ステージング（変更制限）、本番（読み取り専用）
- **AI 生成コードのコミットタグ**: Git co-authorship で追跡可能に

### 核心的原則

> 「AI 生成コードの欠陥に対する責任は AI ではなく人間にある」— タスクを委譲し、出力をレビューし、マージを承認した開発者が失敗の責任を持つ。
