# 新機能実装プロンプトテンプレート

> Phase 2 の調査（10-task-new-feature）に基づく推奨パターン。Phase 4B の実験で検証予定。

## 規模別アプローチ

| 規模 | 目安 | アプローチ |
|------|------|-----------|
| 小 | <200 行、1-2 ファイル | 直接指示 |
| 中 | 200-500 行、3-5 ファイル | Plan Mode → 実装 |
| 大 | 500+ 行、5+ ファイル | 仕様策定 → 設計 → タスク分割 → 複数セッション |

## パターン 1: 小規模（直接指示）

```
`src/utils/` に `formatCurrency` 関数を追加する。

仕様:
- 数値を受け取り、日本円表記の文字列を返す（例: 1234 → "¥1,234"）
- 負の値は "-¥1,234" 形式
- 小数点以下は切り捨て
- NaN や Infinity には "¥0" を返す

テストケース:
- formatCurrency(1234) → "¥1,234"
- formatCurrency(-500) → "-¥500"
- formatCurrency(0) → "¥0"
- formatCurrency(NaN) → "¥0"

テストも一緒に作成し、実行して通ることを確認してください。
```

## パターン 2: 中規模（Plan Mode 活用）

```
ユーザープロフィール編集機能を追加する。

要件:
- /settings/profile でプロフィール編集フォームを表示
- 編集可能: 表示名、自己紹介（500 文字以内）、アバター画像
- バリデーション: 表示名は必須・50 文字以内
- 保存は既存の API エンドポイント PATCH /api/users/:id を使う

制約:
- 既存の `src/components/Form/` のフォームコンポーネントを再利用する
- スタイルは既存の設定画面 `src/pages/settings/` に合わせる

まず Plan Mode で設計を確認させてください。
実装後はフォームのバリデーションテストを書いて実行してください。
```

## パターン 3: 大規模（仕様駆動）

### セッション 1: 仕様策定

```
通知システムを新規追加したい。以下の要件を整理して設計書を作成してください。

ビジネス要件:
- ユーザーにリアルタイム通知を送る（コメント、メンション、ステータス変更）
- 未読/既読管理
- 通知設定（種類ごとの ON/OFF）

技術制約:
- 既存の WebSocket 基盤 `src/lib/ws/` を使う
- DB は PostgreSQL（既存スキーマ: @docs/schema.md 参照）

以下をまとめてください:
1. データモデル設計
2. API エンドポイント一覧
3. フロントエンドコンポーネント構成
4. 実装タスクの分割（各タスク 200 行以内）
```

### セッション 2 以降: タスク単位で実装

```
通知システムの設計書（@docs/notification-design.md）に基づいて、
タスク 1「通知テーブルの作成とモデル定義」を実装してください。

- マイグレーションファイルを作成
- モデル定義を追加
- 基本的な CRUD テストを書いて実行
```

## 6 要素スペックフレームワーク

中〜大規模の機能では、以下の 6 要素を明示すると精度が上がる:

1. **コマンド**: ビルド・テスト・リントの実行方法
2. **テスト**: テストの書き方と実行方法
3. **プロジェクト構造**: 関連するディレクトリとファイル
4. **コードスタイル**: 既存のパターンや規約
5. **Git ワークフロー**: ブランチ・コミットの方針
6. **境界**: やらないこと・変更しない範囲の明示

## 避けるべきパターン

| やりがち | 改善 |
|----------|------|
| 「いい感じの機能を作って」 | 具体的な仕様・入出力を定義 |
| 計画と実装を 1 セッションに詰め込む | セッションを分ける（コンテキスト劣化防止） |
| 500 行超の実装を一度に依頼 | 200 行以内のタスクに分割 |
| 既存パターンを参照しない | `src/pages/existing/` のように具体的パスを示す |
