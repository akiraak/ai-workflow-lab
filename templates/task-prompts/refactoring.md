# リファクタリングプロンプトテンプレート

> Phase 2 の調査（11-task-refactoring）に基づく推奨パターン。Phase 4B の実験で検証予定。

## 基本原則

- **ロジック変更を禁止する**: リファクタリングでは振る舞いを変えない
- **テストで保護する**: 既存テストを先に確認、不足なら特性テストを追加
- **ファイル数を制限する**: 1 回のリファクタリングは 7 ファイル以内
- **Plan Mode を使う**: 影響範囲を把握してから着手

## 基本フォーマット

```
{{リファクタリングの対象と手法を具体的に}}

対象ファイル: {{ファイルパスを列挙、または @ファイル参照}}

制約:
- 外部から見た振る舞いは変更しない
- 既存テストがすべて通ることを確認する

手順:
1. まず既存テストを実行して現在の状態を確認
2. リファクタリング計画を提示（コード変更なし）
3. 承認後に実装
4. テスト・リント・型チェックを実行
```

## パターン別テンプレート

### パターン 1: メソッド抽出

```
`src/services/orderService.ts` の `processOrder` 関数が 150 行ある。
以下の責務に分割してほしい:

- バリデーション → `validateOrder`
- 在庫確認 → `checkInventory`
- 決済処理 → `processPayment`
- メール通知 → `sendConfirmation`

制約:
- `processOrder` の公開インターフェースは変えない
- ロジックの変更は禁止（抽出のみ）
- 既存テスト `src/services/__tests__/orderService.test.ts` が通ること
```

### パターン 2: 非同期変換

```
`src/lib/fileProcessor.ts` のコールバックベースの処理を
async/await に変換する。

対象関数:
- `readConfig` (L12-45)
- `processFiles` (L47-120)
- `writeOutput` (L122-160)

制約:
- エラーハンドリングの振る舞いを維持する
- 呼び出し元の変更は最小限にする
- 変換後にテストを実行して動作を確認
```

### パターン 3: モジュール分割

```
`src/utils/helpers.ts`（800 行）をドメイン別に分割する。

Plan Mode で以下を分析してください:
1. 各関数の利用箇所（依存グラフ）
2. 分割案（ドメインごとのファイル構成）
3. 循環依存が発生しないことの確認

分析結果を確認してから実装に進みます。
```

### パターン 4: TDD インターリーブ

```
`src/models/User.ts` のバリデーションロジックを `src/validators/userValidator.ts` に抽出する。

以下のサイクルで進めてください:
1. 現在の振る舞いを確認する特性テストを書く
2. テストが通ることを確認
3. ロジックを抽出
4. テストが通ることを再確認
5. 次の関数に移る

各ステップの結果を報告してください。
```

## リファクタリング前のチェックリスト

- [ ] 既存テストがあるか確認した
- [ ] テストが不足する場合、特性テストを先に書く指示を含めた
- [ ] 対象ファイルを具体的に指定した（自動探索に頼らない）
- [ ] 「ロジック変更禁止」の制約を明記した
- [ ] 検証手順（テスト・リント・型チェック）を含めた
- [ ] 1 回の対象を 7 ファイル以内に絞った

## 避けるべきパターン

| やりがち | 改善 |
|----------|------|
| 「コードをきれいにして」 | 具体的な手法を指定（メソッド抽出、モジュール分割など） |
| 対象ファイルを指定しない | `@src/services/orderService.ts` のように明示 |
| テスト確認を省略 | テスト実行を必須ステップに含める |
| 10+ ファイルを一度に変更 | 7 ファイル以内に制限、超える場合は分割 |
| ロジック変更と混ぜる | リファクタリングと機能変更は別コミット |
