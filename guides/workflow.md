# 段階的ワークフローガイド

> Phase 1〜3 の調査に基づく推奨ワークフロー。Phase 4B の実験で検証予定。

## 4 フェーズワークフロー

```
Explore → Plan → Code → Commit
```

| フェーズ | モード | 目的 | 主な操作 |
|----------|--------|------|---------|
| Explore | Plan Mode | 現状把握 | ファイル構造の確認、コード読解、依存関係の把握 |
| Plan | Plan Mode | 設計 | 実装方針の策定、影響範囲の特定、タスク分割 |
| Code | Normal Mode | 実装 | コード編集、テスト作成・実行 |
| Commit | Normal Mode | 確定 | 検証チェックリスト実行、コミット |

## タスク規模別の判断基準

### 小規模: 直接実装

**条件**: diff を 1 文で説明できる

- 単一ファイルの修正
- 明確なバグ修正（原因がわかっている）
- 小さな関数の追加
- 設定値の変更

**ワークフロー**: Code → Commit（Explore・Plan を省略）

```
例: 「src/config.ts のタイムアウト値を 3000 → 5000 に変更して」
```

### 中規模: Plan Mode 活用

**条件**: 3〜5 ファイルに影響、または設計判断が必要

- 既存機能の拡張
- 中規模のリファクタリング
- 新しいコンポーネント・エンドポイントの追加

**ワークフロー**: Explore → Plan → Code → Commit

```
例: 「ユーザープロフィール編集機能を追加して。
     まず Plan Mode で設計を見せて」
```

### 大規模: 複数セッション

**条件**: 5+ ファイル、500+ 行の変更、アーキテクチャ判断を含む

- 新しいシステム・サブシステムの構築
- 大規模リファクタリング
- 認証基盤の変更

**ワークフロー**:

```
セッション 1: Explore → Plan（設計書を出力）
  ↓ /clear
セッション 2: Code → Commit（タスク 1）
  ↓ /clear
セッション 3: Code → Commit（タスク 2）
  ...
```

## 各フェーズの詳細

### Explore フェーズ

**目的**: コードベースの現状を把握する

```
Plan Mode で以下を確認してください:
- {{対象領域}} の現在の構造
- 関連するファイルと依存関係
- 既存のテストカバレッジ
- 変更が影響する範囲
```

ポイント:
- `Shift+Tab` で Plan Mode に切り替え
- ファイルパスがわかる場合は `@ファイル名` で直接指定
- 大規模コードベースではサブエージェントに探索を委譲

### Plan フェーズ

**目的**: 実装方針を決定し、合意を取る

```
以下の方針で実装を計画してください:
- 変更するファイル一覧
- 各ファイルの変更内容
- 実装の順序
- リスクや注意点
```

ポイント:
- 最初の計画をそのまま承認しない（2〜3 回の修正サイクルが普通）
- `Ctrl+G` でエディタ上から計画を直接編集できる
- 不明点があれば Claude に質問させる（インタビュー技法）

### Code フェーズ

**目的**: 計画に沿って実装する

```
計画に沿って実装してください。
各ステップ完了後にテストを実行して動作を確認してください。
```

ポイント:
- `Shift+Tab` で Normal Mode に切り替え
- 変更が大きい場合は中間コミットを作る
- コンテキスト使用率が 60% に近づいたら `/compact` で圧縮

### Commit フェーズ

**目的**: 変更を検証して確定する

```
以下を実行してコミットしてください:
1. リント: {{lint command}}
2. テスト: {{test command}}
3. 型チェック: {{type check command}}
4. すべて通ったらコミット
```

## セッション管理のルール

| ルール | 理由 |
|--------|------|
| 1 タスク = 1 セッション | コンテキスト汚染を防ぐ |
| 3 時間以上のセッションは分割 | 長時間セッションで品質が劣化する |
| 同じ問題に 2 回失敗したら `/clear` | 修正の繰り返しは泥沼化する |
| タスク切り替え時は `/clear` | 前のタスクのコンテキストが干渉する |
| コンテキスト 60% で `/compact` | 自動圧縮（95%）まで待つと品質が落ちる |

## 5 分ルール

> 手作業で 5 分以上かかりそうなタスクは Plan Mode を使う。

これは過剰計画と計画不足のバランスを取る簡易判断基準。

- 5 分未満 → 直接実装
- 5 分以上 → Plan Mode で設計してから着手
