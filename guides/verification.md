# 検証手段の設計ガイド

> Phase 1〜3 の調査に基づく推奨。Phase 4B の実験で検証予定。

## 検証の重要性

> 「テストケース、スクリーンショット、期待される出力を提供することが、Claude Code を使う上で最もレバレッジの高い行動」
> — Anthropic 公式ドキュメント

検証手段を指定しないと、Claude は「動くコード」を書いても「正しいコード」を書いているとは限らない。

## 検証レベル

| レベル | 名称 | 内容 | 適用場面 |
|--------|------|------|---------|
| Level 0 | なし | 検証なし | 使わない（非推奨） |
| Level 1 | 静的検証 | リント + 型チェック | 設定変更、フォーマット修正 |
| Level 2 | ユニットテスト | テスト作成・実行 | 関数追加、ロジック修正 |
| Level 3 | 統合テスト | 複数コンポーネントのテスト | API エンドポイント、データフロー |
| Level 4 | E2E + 手動確認 | ブラウザテスト、スクリーンショット | UI 変更、ユーザーフロー |

## タスク種別ごとの推奨検証レベル

### バグ修正

```
推奨: Level 2〜3（TDD パターン）

検証手順:
1. バグを再現する失敗テストを書く         ← 問題の証明
2. テストが失敗することを確認             ← テスト自体の検証
3. 最小限のコード変更で修正               ← 修正
4. テストが通ることを確認                 ← 修正の証明
5. 既存テスト全体を実行                   ← 回帰なしの証明
```

**プロンプト例**:
```
修正後、以下を実行して結果を報告してください:
1. npm test -- --grep "validateToken" （対象テスト）
2. npm test （全体テスト）
3. npm run lint
```

### 新機能実装

```
推奨: Level 2〜4（規模による）

小規模（Level 2）:
1. 関数のユニットテストを作成・実行
2. リント + 型チェック

中規模（Level 3）:
1. ユニットテスト作成・実行
2. 統合テスト作成・実行（API エンドポイント等）
3. リント + 型チェック

大規模（Level 4）:
1. ユニットテスト + 統合テスト
2. E2E テスト（主要フロー）
3. スクリーンショット比較（UI 変更がある場合）
4. リント + 型チェック
```

### リファクタリング

```
推奨: Level 2〜3（振る舞い不変の証明）

検証手順:
1. 既存テストを実行（ベースライン）       ← 変更前の状態を記録
2. テストが不足していれば特性テストを追加   ← 保護を強化
3. リファクタリング実施
4. 全テスト再実行                         ← 振る舞い不変の証明
5. リント + 型チェック                    ← 構文・型の整合性
```

**プロンプト例**:
```
リファクタリング前後で以下が一致することを確認してください:
1. npm test の結果（全テスト PASS）
2. npx tsc --noEmit の結果（型エラーなし）
3. npm run lint の結果（リントエラーなし）
```

### コードレビュー

```
推奨: Level 1〜2（指摘の裏付け）

検証手順:
1. 指摘した問題を再現するテストケースを示す  ← 指摘の証明
2. 修正案が既存テストを壊さないことを確認    ← 修正の安全性
```

## 検証指示のテンプレート

### 最小限の検証（Level 1）

```
変更後、以下を実行してください:
- `npm run lint`
- `npx tsc --noEmit`
```

### 標準の検証（Level 2）

```
変更後、以下を実行してください:
1. 対象のテストを実行: `npm test -- {{test file}}`
2. 全テストを実行: `npm test`
3. リント: `npm run lint`
4. 型チェック: `npx tsc --noEmit`

すべて通ることを確認してからコミットしてください。
```

### 厳格な検証（Level 3-4）

```
変更後、以下の検証を段階的に実行してください:

Step 1 - ユニットテスト:
  npm test -- {{test file}}

Step 2 - 統合テスト:
  npm run test:integration

Step 3 - E2E テスト（該当する場合）:
  npm run test:e2e -- --grep "{{関連シナリオ}}"

Step 4 - 静的解析:
  npm run lint && npx tsc --noEmit

各ステップの結果を報告してください。
失敗がある場合は修正してから次に進んでください。
```

## テスト改ざんへの対策

Claude は既存テストを修正して「テストを通す」ことがある（アンチパターン）。

### 防止策

1. **CLAUDE.md にルールを書く**:
   ```markdown
   既存テストの期待値やアサーションを変更しない。
   テストが失敗する場合は実装側を修正する。
   ```

2. **Hooks で保護する**:
   ```json
   {
     "hooks": {
       "PreEdit": [{
         "matcher": "**/*.test.*",
         "command": "echo '⚠️ テストファイルの編集には注意'"
       }]
     }
   }
   ```

3. **レビューセッションで確認**:
   テスト変更がある場合は、別セッションでレビュー

## 検証レベルの選択フローチャート

```
設定変更やフォーマット修正？
├─ Yes → Level 1（リント + 型チェック）
└─ No
    ├─ 単一関数の追加・修正？
    │   └─ Yes → Level 2（ユニットテスト）
    ├─ 複数コンポーネントにまたがる変更？
    │   └─ Yes → Level 3（統合テスト）
    └─ UI やユーザーフローの変更？
        └─ Yes → Level 4（E2E + スクリーンショット）
```
